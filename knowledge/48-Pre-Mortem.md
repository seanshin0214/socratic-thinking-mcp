# Pre-Mortem Analysis (사전부검)

## 개요
Pre-Mortem은 프로젝트가 "실패했다"고 가정하고 그 원인을 역추적하는 기법입니다. 일반적인 리스크 분석과 달리, 실패를 기정사실로 놓고 "왜 실패했는가?"를 묻습니다. 심리학자 Gary Klein이 개발했습니다.

## 핵심 원리

### Post-Mortem vs Pre-Mortem
```
Post-Mortem (사후부검)           Pre-Mortem (사전부검)
─────────────────────           ─────────────────────
프로젝트 완료 후                 프로젝트 시작 전
"왜 실패했는가?"                 "왜 실패할 것인가?"
이미 늦음                        예방 가능
책임 추궁 분위기                 안전한 상상
```

### Prospective Hindsight (전망적 사후통찰)
```
일반 질문: "이 프로젝트가 실패할 수 있을까?"
→ 낙관적 응답, 표면적 리스크만 언급

Pre-Mortem 질문: "이 프로젝트는 실패했다. 왜?"
→ 구체적 실패 원인, 숨겨진 우려 표출
→ 30% 더 많은 리스크 식별 (연구 결과)
```

## Pre-Mortem 진행 방법

### 단계 1: 설정 (5분)
```
1. 팀을 모읍니다
2. 프로젝트/결정을 간단히 설명
3. 규칙 안내:
   - 실패는 확정된 사실
   - 어떤 아이디어도 환영
   - 비난 없음
   - 솔직하게
```

### 단계 2: 실패 상상 (2분)
```
퍼실리테이터:
"지금은 [프로젝트 완료 예정일]입니다.
이 프로젝트는 완전히 실패했습니다.
대재앙입니다.
잠시 눈을 감고 그 상황을 상상해 보세요."
```

### 단계 3: 개인 작성 (5-10분)
```
각자 포스트잇에 작성:
"이 프로젝트가 실패한 이유는..."

규칙:
- 한 장에 하나의 이유
- 최소 3개 이상
- 구체적으로
- 이름 안 씀 (익명)
```

### 단계 4: 공유 (10-15분)
```
방법 1: 라운드 로빈
- 한 명씩 하나의 이유 발표
- 중복은 그룹화
- 모든 사람이 발표할 때까지

방법 2: 조용한 분류
- 모든 포스트잇을 벽에 붙임
- 조용히 읽고 유사한 것 그룹화
- 그룹별 토론
```

### 단계 5: 우선순위 (5-10분)
```
기준:
- 발생 가능성 (높음/중간/낮음)
- 영향력 (치명/심각/경미)

매트릭스:
           영향력
      │ 높음  중간  낮음
──────┼─────────────────
높음  │  🔴   🟠   🟡
가능성 중간  │  🟠   🟡   🟢
낮음  │  🟡   🟢   🟢
```

### 단계 6: 대응 계획 (10-15분)
```
각 고위험 항목에 대해:

실패 원인: _____________

예방 조치:
□ 조치 1: _________ (담당: ___, 기한: ___)
□ 조치 2: _________ (담당: ___, 기한: ___)

조기 경고 신호:
- 신호 1: _________
- 신호 2: _________

대응 계획 (발생 시):
- 계획 A: _________
- 계획 B: _________
```

## Pre-Mortem 질문 프롬프트

### 팀/인력 관련
```
- 핵심 인력이 떠났다면?
- 팀 갈등이 해결되지 않았다면?
- 필요한 기술이 부족했다면?
- 커뮤니케이션이 단절되었다면?
```

### 기술/실행 관련
```
- 기술적 장애물을 넘지 못했다면?
- 일정이 지연되었다면?
- 품질 문제가 발생했다면?
- 의존성이 문제가 되었다면?
```

### 외부 요인 관련
```
- 시장이 변했다면?
- 경쟁자가 먼저 출시했다면?
- 규제가 바뀌었다면?
- 경제 상황이 악화되었다면?
```

### 내부 정치 관련
```
- 경영진 지원이 사라졌다면?
- 예산이 삭감되었다면?
- 우선순위가 변경되었다면?
- 핵심 이해관계자가 반대했다면?
```

## 예시: 신제품 출시 Pre-Mortem

### 상황
- 프로젝트: 새로운 모바일 앱 6개월 내 출시
- 팀: 개발 5명, 디자인 2명, PM 1명

### Pre-Mortem 결과

```
식별된 실패 원인:

[높은 가능성 + 높은 영향]
1. 핵심 개발자 퇴사 → 지식 손실
   예방: 문서화 강화, 페어 프로그래밍

2. 범위 확대 (Scope Creep)
   예방: 엄격한 변경 관리, MVP 명확화

[높은 가능성 + 중간 영향]
3. 테스트 시간 부족 → 버그 많은 출시
   예방: 자동화 테스트, 베타 기간 확보

4. 마케팅 협업 지연
   예방: 마케팅 조기 참여, 정기 미팅

[중간 가능성 + 높은 영향]
5. 앱스토어 심사 거절
   예방: 가이드라인 사전 검토, 버퍼 일정
```

## Pre-Mortem 변형

### 성공 Pre-Mortem
```
"프로젝트가 대성공했습니다. 왜?"
- 성공 요인 식별
- 강화해야 할 것 파악
- 긍정적 분위기 유지
```

### Red Team Pre-Mortem
```
일부 팀원이 "적"이 되어:
- 경쟁자 관점에서 우리 약점 공격
- 규제 기관 관점에서 문제 지적
- 불만 고객 관점에서 비판
```

### Quick Pre-Mortem (10분)
```
1. 실패 상상 (1분)
2. 각자 Top 3 원인 작성 (3분)
3. 공유 및 투표 (3분)
4. 즉시 조치 1가지 결정 (3분)
```

## Pre-Mortem 효과

### 인지 편향 극복
```
- 확증 편향: "우리 계획은 좋아" → 약점 직시
- 낙관 편향: "잘 될 거야" → 현실적 평가
- 집단사고: "다 같은 생각" → 이견 표출
- 계획 오류: "시간 충분해" → 리스크 고려
```

### 팀 역학 개선
```
- 솔직한 의견 표출 안전하게
- 주니어도 우려 표현 가능
- 비난 없이 문제 제기
- 공동 책임감 형성
```

## 체크리스트

### Pre-Mortem 전
```
□ 프로젝트 개요 문서 준비
□ 참석자 전원 초대 (다양한 역할)
□ 회의실/화이트보드 준비
□ 포스트잇, 마커
□ 타이머
```

### Pre-Mortem 중
```
□ 안전한 분위기 조성
□ 모든 사람 발언 기회
□ 아이디어 기록
□ 시간 관리
□ 우선순위화
```

### Pre-Mortem 후
```
□ 결과 문서화
□ 조치 항목 담당자 배정
□ 정기 검토 일정
□ 프로젝트 계획에 반영
```
