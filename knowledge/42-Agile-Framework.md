# Agile Framework (애자일 프레임워크)

## 개요
애자일은 복잡하고 불확실한 환경에서 가치를 빠르게 전달하기 위한 반복적, 점진적 개발 철학입니다. 2001년 17명의 소프트웨어 개발자가 "애자일 선언문"을 발표하며 시작되었습니다.

## 애자일 4가지 핵심 가치

```
┌────────────────────────────────────────┐
│         애자일 선언문 (2001)           │
├────────────────────────────────────────┤
│ 프로세스와 도구보다 → 개인과 상호작용  │
│ 포괄적 문서보다    → 작동하는 소프트웨어│
│ 계약 협상보다      → 고객과 협력       │
│ 계획을 따르기보다  → 변화에 대응       │
└────────────────────────────────────────┘
```

## 애자일 12가지 원칙

1. 가치 있는 소프트웨어를 일찍, 지속적으로 전달
2. 요구사항 변경을 환영
3. 작동하는 소프트웨어를 자주 전달 (주~월 단위)
4. 비즈니스와 개발자가 매일 함께 일함
5. 동기 부여된 개인들로 프로젝트 구성
6. 면대면 대화가 가장 효과적
7. 작동하는 소프트웨어가 진척의 주된 척도
8. 지속 가능한 개발 속도 유지
9. 기술적 탁월함과 좋은 설계에 지속 관심
10. 단순함 - 하지 않는 일의 양을 최대화
11. 자기조직화 팀에서 최고의 설계 탄생
12. 정기적으로 효율성 향상 방법을 회고

## Scrum 프레임워크

### 역할 (Roles)

#### Product Owner (PO)
- 제품 비전 정의
- 백로그 관리 및 우선순위 결정
- 이해관계자와 소통
- 가치 최대화 책임

#### Scrum Master (SM)
- 스크럼 프로세스 촉진
- 장애물 제거
- 팀 보호 및 코칭
- 조직 변화 지원

#### Development Team
- 자기조직화
- 크로스펑셔널 (모든 기술 보유)
- 5-9명 권장
- 제품 증분 전달 책임

### 산출물 (Artifacts)

#### Product Backlog
```
우선순위   사용자 스토리              추정
────────────────────────────────────────
  1      사용자가 로그인할 수 있다     5
  2      프로필 사진을 업로드한다      3
  3      비밀번호를 재설정한다         2
  4      소셜 로그인을 한다            8
  ...
```

#### Sprint Backlog
- 이번 스프린트에서 수행할 항목
- 팀이 커밋한 작업
- 매일 업데이트

#### Product Increment
- 스프린트 결과물
- "완료" 정의 충족
- 잠재적으로 출시 가능

### 이벤트 (Events)

#### Sprint (1-4주)
- 고정된 기간의 반복 주기
- 2주가 가장 일반적
- 스프린트 목표 설정

#### Sprint Planning (스프린트 계획)
- **시간**: 4시간 (2주 스프린트 기준)
- **참석**: 전체 스크럼 팀
- **목적**: 스프린트 목표와 백로그 선정

#### Daily Standup (일일 스탠드업)
- **시간**: 15분
- **참석**: 개발팀 (+ SM, PO 선택)
- **질문**:
  1. 어제 무엇을 했는가?
  2. 오늘 무엇을 할 것인가?
  3. 장애물이 있는가?

#### Sprint Review (스프린트 리뷰)
- **시간**: 2시간 (2주 스프린트 기준)
- **참석**: 스크럼 팀 + 이해관계자
- **목적**: 증분 시연 및 피드백

#### Sprint Retrospective (스프린트 회고)
- **시간**: 1.5시간 (2주 스프린트 기준)
- **참석**: 스크럼 팀
- **목적**: 프로세스 개선

## 회고 기법

### Start-Stop-Continue
```
┌─────────────┬─────────────┬─────────────┐
│   Start     │    Stop     │  Continue   │
│   시작할 것  │  그만할 것   │  계속할 것  │
├─────────────┼─────────────┼─────────────┤
│ 페어 프로그래밍│ 긴 회의     │ 일일 스탠드업│
│ 코드 리뷰    │ 야근        │ 슬랙 활용   │
└─────────────┴─────────────┴─────────────┘
```

### 4Ls (Liked, Learned, Lacked, Longed for)
- **Liked**: 좋았던 것
- **Learned**: 배운 것
- **Lacked**: 부족했던 것
- **Longed for**: 바라는 것

### Mad Sad Glad
- 감정 기반 회고
- 팀 분위기 파악에 효과적

## Kanban 보드

```
┌──────────┬──────────┬──────────┬──────────┐
│ To Do    │ In Prog  │ Review   │ Done     │
│          │ (WIP: 3) │ (WIP: 2) │          │
├──────────┼──────────┼──────────┼──────────┤
│ [Task A] │ [Task D] │ [Task G] │ [Task J] │
│ [Task B] │ [Task E] │ [Task H] │ [Task K] │
│ [Task C] │ [Task F] │          │          │
└──────────┴──────────┴──────────┴──────────┘
        WIP = Work In Progress Limit
```

### Kanban 원칙
1. 현재 프로세스에서 시작
2. 점진적, 진화적 변화 추구
3. 현재 역할과 책임 존중
4. 리더십을 모든 레벨에서 장려

### Kanban 실천
1. 작업 시각화
2. WIP 제한
3. 흐름 관리
4. 프로세스 정책 명시화
5. 피드백 루프 구현
6. 협력적 개선

## 사용자 스토리 작성

### 형식
```
As a [사용자 유형]
I want [기능/행동]
So that [가치/이유]

예시:
As a 신규 사용자
I want 이메일로 가입할 수 있기를
So that 서비스를 이용할 수 있다
```

### INVEST 기준
- **I**ndependent: 독립적
- **N**egotiable: 협상 가능
- **V**aluable: 가치 있는
- **E**stimable: 추정 가능
- **S**mall: 작은
- **T**estable: 테스트 가능

## 추정 기법

### Planning Poker
1. 각 팀원에게 피보나치 카드 배포 (1, 2, 3, 5, 8, 13, 21...)
2. 스토리 설명
3. 동시에 카드 공개
4. 차이가 크면 토론 후 재투표

### T-Shirt Sizing
- XS, S, M, L, XL로 상대적 크기 추정
- 빠른 추정에 적합

## 스크럼 vs 칸반 vs XP

| 측면 | Scrum | Kanban | XP |
|------|-------|--------|-----|
| 반복 주기 | 고정 (1-4주) | 연속적 흐름 | 1-2주 |
| 역할 | PO, SM, Team | 없음 | Coach, Customer |
| 변경 | 스프린트 내 제한 | 언제든 가능 | 유연 |
| 적합 | 제품 개발 | 운영/유지보수 | 소프트웨어 |
| WIP 제한 | 스프린트 용량 | 열(column)별 | 없음 |

## 애자일 도입 실패 요인

1. **경영진 지원 부족**: 변화에 대한 저항
2. **부분 도입**: 일부만 적용하고 효과 기대
3. **문화 무시**: 프랙티스만 도입, 마인드셋 무시
4. **무리한 일정**: 스크럼 ≠ 더 빠름
5. **회고 생략**: 개선 없이 반복
6. **PO 부재**: 의사결정 지연

## 확장 프레임워크

### SAFe (Scaled Agile Framework)
- 대기업용 애자일 확장
- 팀, 프로그램, 포트폴리오 레벨

### LeSS (Large-Scale Scrum)
- 스크럼의 단순한 확장
- 하나의 백로그, 하나의 PO

### Spotify Model
- Squads, Tribes, Chapters, Guilds
- 자율성과 정렬의 균형
